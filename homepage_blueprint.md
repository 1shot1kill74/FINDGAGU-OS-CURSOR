# Homepage Blueprint

## 1. 문서 목적

이 문서는 파인드가구 공개 홈페이지를 현재 `FindGagu OS`와 연동하여 재구성하기 위한 실무용 설계 문서다.

핵심 방향은 다음 한 줄로 정리한다.

**이미지 자산 관리 시스템이 홈페이지 콘텐츠 엔진이 된다.**

즉, 홈페이지를 별도 외부 툴에서 수동 편집하는 구조가 아니라, 현재 OS 안의 `image_assets`와 공개 동선을 중심으로 운영 가능한 구조로 전환한다.

이번 개편의 운영 철학은 다음으로 고정한다.

**컨설팅처럼 느껴지되, 대표가 직접 많이 개입하지 않아도 돌아가는 저관여 고신뢰 구조를 만든다.**

---

## 2. 현재 확인된 팩트

### 2-1. 현재 OS 안에 이미 존재하는 공개 동선

현재 코드베이스에는 아래 공개 동선이 이미 존재한다.

- `image_assets` 기반 공개 사례 조회
- `/showroom` 공개 쇼룸 페이지
- `/contact` 공개 문의 페이지
- 사례 클릭 후 문의 맥락 전달

즉, "홈페이지 기능을 처음부터 새로 만드는 상태"는 아니다.  
공개 랜딩으로 확장 가능한 뼈대가 이미 존재한다.

### 2-2. 현재 기술 스택

- Frontend: Vite + React + TypeScript
- Router: React Router
- Backend/DB: Supabase
- 이미지: Cloudinary + Supabase Storage 이원화

### 2-3. 현재 데이터 소스

홈페이지/쇼룸의 핵심 데이터 소스 후보는 `image_assets`다.

현재 `image_assets`에서 활용 가능한 주요 필드는 다음과 같다.

- `site_name`
- `location`
- `business_type`
- `product_name`
- `color_name`
- `is_main`
- `created_at`
- `metadata.before_after_role`
- `metadata.before_after_group_id`

### 2-4. 현재 시스템의 공개 전환 구조

현재 구조상 다음 흐름은 이미 가능하다.

1. 이미지 자산 등록
2. 쇼룸에서 사례 노출
3. 사례별 맥락을 붙여 문의 페이지 진입
4. 문의 메타데이터 저장

즉, 쇼룸은 단순 갤러리가 아니라 `사례 -> 문의` 전환 퍼널의 시작점이다.

## 3. 홈페이지 재구성의 기본 판단

### 3-1. 전체 마이그레이션은 하지 않는다

기존 데이터를 "다 올리는 것"을 목표로 하지 않는다.

우선순위는 아래와 같다.

- 상담 중 바로 쓸 수 있는가
- 업종/현장/제품 태깅이 가능한가
- 대표컷이 강한가
- 현장 이해가 가능한가
- 주력 상품이 보이는가
- 리뉴얼이면 설득 포인트가 있는가
- 가능하면 Before/After가 있는가

즉, **전체 이전보다 선별 이전이 우선**이다.

### 3-2. 홈페이지의 주어는 현장으로 둔다

기존 홈페이지는 제품, 사례, 콘텐츠가 혼합된 구조였다.  
새 구조에서는 메인 서사를 다음처럼 바꾼다.

- 기존: 제품이 메인, 현장은 증거
- 새 구조: 현장이 메인, 제품은 결과를 만드는 솔루션

단, 제품 페이지나 제품 정보가 사라지는 것은 아니다.  
메인 홈페이지에서의 우선순위만 조정한다.

### 3-3. 목표는 "많이 했다"가 아니라 "진짜 잘한다"를 보여주는 것

대표 노출 현장은 많을 필요가 없다.  
중요한 것은 아래다.

- 결과물이 강한가
- 설명이 쉬운가
- 고객이 자기 공간을 상상할 수 있는가
- 실제 문의로 이어질 가능성이 높은가

따라서 사진 수가 적고 설득 포인트가 약한 현장은 메인 노출 대상에서 제외할 수 있다.

### 3-4. 컨설팅 회사 분리 전략보다 "저관여 고신뢰 구조"가 우선이다

이 문서는 별도 공간 컨설팅 회사를 새로 만드는 전략을 기본안으로 두지 않는다.

현재 더 적합한 방향은 아래다.

- 고객에게는 전문성과 기준이 느껴진다
- 홈페이지와 사례가 먼저 많은 설명을 대신한다
- 직원이 표준화된 흐름으로 1차 응대를 수행한다
- AI가 분류, 추천, 초안 작성을 보조한다
- 대표는 예외적인 고가치 건에만 제한적으로 개입한다

즉, 목표는 "대표가 더 많이 붙는 컨설팅 회사"가 아니라,  
**대표의 판단 기준이 시스템과 자산에 스며든 회사**를 만드는 것이다.

---

## 4. 홈페이지의 역할 정의

새 홈페이지는 단순 회사 소개 페이지가 아니라 아래 5가지 역할을 수행해야 한다.

1. 업종별 대표 사례를 보여주는 공개 랜딩
2. 사례를 탐색하게 하는 쇼룸 진입점
3. 상담 전환을 만드는 퍼널
4. 직원이 설명할 내용을 먼저 대신 안내하는 1차 설명 인터페이스
5. 향후 블로그/유튜브/AI 검색에 재사용되는 콘텐츠 허브

즉, 홈페이지는 "디자인 결과물"이 아니라 "공개 콘텐츠 운영 인터페이스"이자  
**대표 시간을 절약하는 1차 설명 시스템**에 가깝다.

---

## 5. 공개 홈페이지 정보 구조(IA)

### 5-0. 메인 홈페이지의 운영 역할

메인 홈페이지는 진짜 컨설팅을 수행하는 곳이 아니다.  
메인 홈페이지의 역할은 아래와 같다.

- 고객이 자기 업종과 비슷한 사례를 먼저 보게 한다
- 제품 가격 비교보다 공간 결과와 기준에 먼저 집중하게 한다
- 직원이 반복 설명하던 내용을 먼저 이해시키게 한다
- 상담 전에 고객 문맥을 어느 정도 정리하게 한다

따라서 메인 홈페이지는 "대표 상담 유치 페이지"가 아니라,  
**비딩을 줄이고 설명 비용을 줄이는 구조적 장치**로 설계한다.

### 5-1. 메인 홈 `/`

메인 홈은 새로 만들되, 현재 OS의 공개 자산과 연동해서 구성한다.

권장 섹션:

1. 히어로
2. 진입 분기 섹션 (`현장부터 보기` / `제품부터 보기`)
3. 고민 선택 섹션
4. 업종별 대표 현장
5. Before/After 또는 리뉴얼 설득 섹션
6. 주력 상품 연결 섹션
7. 상담 CTA

### 5-2. 쇼룸 `/showroom`

쇼룸은 "깊게 탐색하는 사례 아카이브"로 둔다.

현재 구조를 유지하되 아래 방향으로 다듬는다.

- 현장별 탐색
- 업종별 탐색
- 제품별 탐색
- 고민/상황별 탐색
- Before/After 중심 묶음 강조
- 사례별 문의 연결 강화

### 5-3. 문의 `/contact`

문의 페이지는 단순 폼이 아니라 사례 맥락을 이어받는 전환 페이지로 유지한다.

유지 원칙:

- 어떤 현장을 보고 들어왔는지 저장
- 어떤 카테고리로 유입됐는지 저장
- 어떤 상황/고민 문맥인지 저장
- 상담 카드에서 다시 활용 가능해야 함

## 6. 현재 시스템과의 연동 원칙

### 6-1. 데이터 원천

공개 홈페이지의 기본 콘텐츠 원천은 `image_assets`다.

홈페이지에 필요한 대표 데이터는 현재 필드 기준으로 먼저 구성한다.

- 현장명: `site_name`
- 지역: `location`
- 업종: `business_type`
- 제품명: `product_name`
- 색상: `color_name`
- 대표 이미지 여부: `is_main`
- Before/After: `metadata.before_after_role`, `metadata.before_after_group_id`

### 6-2. 공개 노출 기준

기본 원칙:

- 발주서, 평면도 등 내부 운영용 자료는 공개 노출 제외
- 시공 사례 사진 위주로 운영
- 상담용/세일즈용으로 실제 활용 가능한 자산 우선

### 6-3. 현재 재사용할 페이지

기존 구현을 최대한 재사용한다.

- `src/pages/ShowroomPage.tsx`
- `src/pages/ContactPage.tsx`
- `src/lib/imageAssetService.ts`
- `src/App.tsx`

즉, 완전 신규 플랫폼보다는 **현재 OS 공개 동선 고도화**가 기본 전략이다.

### 6-4. AI와 자동화의 역할

AI와 자동화는 대표를 대체하는 용도가 아니라,  
대표가 반복적으로 하던 1차 설명과 분류를 줄이는 방향으로 사용한다.

우선 적용 가능한 역할:

- 업종/상황별 사례 추천 보조
- 문의 문맥 자동 세팅
- 상담 메모 초안 생성
- 직원 응대용 추천 사례/제품 묶음 보조

즉, AI는 "대표 개입 확대 장치"가 아니라  
**직원 대응 품질을 일정 수준으로 끌어올리는 보조 장치**로 본다.

## 7. 자산 선별 원칙

### 7-1. 우선 선별 대상

아래 조건을 많이 만족할수록 우선순위를 높인다.

- Before가 있다
- 결과가 잘 나왔다
- 주력 상품이 보인다
- 업종 대표성이 있다
- 리뉴얼 설득 포인트가 분명하다
- 사진 수가 최소 기준을 충족한다

### 7-2. 현장당 운영용 사진 수 기준

운영용 선별본 기준 권장 수량:

- 메인 랜딩용: 3~5장
- 쇼룸 상세용: 6~12장
- 일반 선별본: 5~8장
- 리뉴얼 설득형: Before/After 포함 4~8장

핵심은 원본 전체 보관이 아니라, **운영용 선별본을 압축해서 만드는 것**이다.

### 7-3. 대표 노출에서 제외해도 되는 경우

아래 현장은 메인 공개 노출 우선순위를 낮춘다.

- 사진 수가 너무 적다
- 대표컷이 약하다
- 주력 상품이 잘 안 보인다
- 공간 이해가 어렵다
- 리뉴얼 맥락이 부족하다
- 유사한 더 강한 현장이 이미 있다

즉, "있으니까 올린다"가 아니라 "쓸 가치가 있는지"를 기준으로 본다.

## 8. 제품과 현장의 관계 정의

새 홈페이지에서 기본 논리는 아래로 정리한다.

- 고객은 먼저 결과를 본다
- 결과를 구성하는 논리로 제품을 보여준다
- 제품은 현장을 설명하는 솔루션으로 배치한다

표현 기준:

- 메인: 현장 중심
- 상세: 현장 중심 + 제품 연결
- 제품 페이지: 제품 중심

즉, 제품은 중요하지만 메인 홈페이지의 주어는 현장이다.

단, 메인 홈에서는 사용자가 처음부터 한 방향만 강요받지 않도록  
`현장부터 보기`와 `제품부터 보기`의 이중 진입 구조를 허용한다.

이때 운영 원칙은 다음과 같다.

- 현장 진입은 결과와 신뢰를 먼저 보여주는 용도
- 제품 진입은 비교보다 적용 맥락을 보여주는 용도
- 두 진입 모두 최종적으로는 사례와 상담으로 연결되어야 한다

---

## 9. AI 검색/검색 친화 구조 방향

### 9-1. 유리한 점

현재 구조는 아래 조건 때문에 향후 검색 친화 구조로 발전시킬 여지가 있다.

- 업종/지역/현장/제품 메타데이터 존재
- 이미지 alt/base text 정책 존재
- 시공 완료 후 자산을 지속적으로 추가할 수 있음

### 9-2. 아직 필요한 조건

단순히 이미지가 많다고 AI 검색에 유리해지지는 않는다.  
아래가 추가로 필요하다.

- 현장별 읽을 만한 텍스트 설명
- 업종별 랜딩 페이지
- 의미 있는 URL 구조
- 메타 태그, OG, sitemap 등 정리
- 공개 페이지 크롤링 친화성 확보

### 9-3. 렌더링 전략

현재는 Vite + React CSR 구조다.  
초기 공개는 현재 구조로도 가능하지만, 향후 검색/AI 검색 최적화가 중요해지면 아래를 검토한다.

- Next.js SSR/SSG 전환
- 또는 공개 페이지 일부만 정적/서버 렌더링 구조로 분리

즉, SSR은 지금 당장 필수는 아니지만 중장기 검토 항목이다.

## 10. 개발 단계 제안

### 1단계. 대표 자산 선별

- 전체 이전 금지
- 시공 사례 위주 선별
- 대표 현장 / 업종별 대표 사례 / Before-After 우선

### 2단계. `image_assets` 메타 정리

- 현장명
- 지역
- 업종
- 제품명
- 색상
- 대표 이미지 여부
- Before/After 연결

### 3단계. `/showroom` 고도화

- 공개 사용성 개선
- 대표 사례 노출 우선순위 조정
- 상담 CTA 강화
- 링크 공유/진입 문맥 정리
- 직원이 바로 활용할 수 있는 사례 설명 흐름 정리

### 4단계. `/` 메인 홈 개편

- 현재 링크 허브형 홈 제거
- 공개 랜딩형 메인 홈 구축
- `현장부터 보기` / `제품부터 보기` 이중 진입 반영
- 쇼룸과 문의로 자연스럽게 연결
- 대표 직접 개입을 전제로 하지 않는 설명 구조 설계

### 5단계. 문의 및 채널 연결 정리

- `/contact` 고도화
- 채널톡/폼 연결 설계
- 유입 경로 및 상담 메타데이터 강화
- 직원 1차 대응과 대표 예외 개입 기준 분리

### 6단계. 검색 구조 보강

- 업종별 공개 페이지
- 사례 설명 텍스트 보강
- SEO/OG/크롤링 구조 개선
- 필요 시 SSR 검토

## 11. 당장 결정된 원칙

현재 시점에서 아래 원칙은 실무 기준으로 확정한다.

1. 홈페이지는 현재 OS와 연동해서 개발한다.
2. 전체 자산 이전보다 선별 이전을 우선한다.
3. 메인 홈페이지의 주어는 제품보다 현장에 둔다.
4. 대표 사례는 "많음"보다 "설득력"을 기준으로 고른다.
5. 쇼룸과 문의 동선은 현재 구현을 재사용한다.
6. 공개 홈페이지는 장기적으로 OS 기반 콘텐츠 엔진 역할을 해야 한다.
7. 홈페이지는 대표의 직접 컨설팅을 늘리는 방향이 아니라, 설명과 분류를 먼저 수행하는 방향으로 설계한다.
8. AI와 자동화는 대표를 더 바쁘게 만드는 것이 아니라, 직원 대응 품질과 운영 효율을 높이는 용도로 사용한다.

## 12. 바로 이어서 해야 할 실무 작업

### 자산 쪽

- 대표 현장 후보 리스트업
- Before/After 보유 현장 우선 분류
- 주력 상품 노출이 좋은 현장 체크
- 사진 수 부족 현장 보류 처리

### 개발 쪽

- 메인 홈 설계안 작성
- 이중 진입형(`현장부터 보기` / `제품부터 보기`) 홈 구조 검토
- 쇼룸 공개 동선 정리
- 문의 메타데이터 확장 필요 여부 검토
- 공개/내부 페이지 분리 정책 정리
- 직원 1차 대응용 사례 추천 흐름 정의

### 운영 쪽

- 홈페이지 메인에 올릴 업종 우선순위 확정
- 주력 상품군 우선순위 확정
- 상담 유입 시 가장 자주 쓰는 대표 사례 확정
- 대표 개입이 필요한 문의 조건 정의

## 13. 최종 정리

이번 홈페이지 개편은 "새 예쁜 사이트 제작"이 아니다.

이번 작업의 본질은 아래다.

- 시공 이미지 자산을 운영 가능한 구조로 선별하고
- 그 자산을 현재 OS 안에서 공개 동선과 연결하고
- 쇼룸, 문의, 상담, 향후 블로그/유튜브/검색까지 이어지는
- 하나의 콘텐츠 운영 시스템으로 만드는 것
- 그리고 대표가 반복 설명하던 기준을 홈페이지, 자산, 직원 응대, AI 보조 흐름에 녹여 넣는 것

따라서 홈페이지 개발은 디자인 작업이 아니라,  
**현재 시스템과 연결된 공개 콘텐츠 엔진 구축 작업**이며,  
동시에 **저관여 고신뢰 운영 구조를 만드는 작업**으로 정의한다.

## 14. AI 오케스트레이션 운영안

### 14-1. 왜 오케스트레이션이 필요한가

홈페이지 개편은 단순 프론트 작업이 아니다.  
현재 과업은 아래 작업이 동시에 얽혀 있다.

- 브랜드 포지셔닝 정리
- 홈페이지 구조 설계
- 현재 OS와의 연동
- 공개 배포/검색 구조 검토
- 이미지 자산 선별 및 메타 정리

이 모든 작업을 하나의 AI나 하나의 세션에서 처리하면 품질이 흔들리기 쉽다.  
따라서 이 프로젝트는 **역할별 AI 분업과 오케스트레이션**을 기본 운영 방식으로 둔다.

핵심 원칙:

- 전략과 구현을 분리한다
- 공개 홈페이지와 내부 OS 관점을 분리한다
- 자산 선별과 프론트 구현을 분리한다
- 최종 의사결정은 대표가 하되, 정보 수집과 초안 생성은 AI가 분담한다

### 14-2. 기본 역할 분리

#### A. 전략 에이전트

역할:

- 홈페이지 방향과 철학 정리
- 제품 중심 vs 현장 중심 판단
- 카피 톤과 진입 구조 검토
- 대표 개입 최소화 원칙 점검

산출물:

- 구조안
- 카피 방향
- 우선순위 문서
- 의사결정 기준표

주의:

- 직접 구현보다 판단 기준 정리에 집중
- 감각적 제안과 확정 사실을 반드시 구분

#### B. 프론트 구현 에이전트

역할:

- 실제 페이지 구현
- 컴포넌트 구성
- 반응형 레이아웃 정리
- 공개용 페이지 품질 개선

산출물:

- 페이지 컴포넌트
- 라우팅 연결
- UI 수정안
- 실제 동작 가능한 프로토타입

주의:

- OS 내부 로직을 함부로 건드리지 말 것
- 공개 홈페이지용 코드와 내부 운영 코드를 혼합하지 말 것

#### C. 연동 검증 에이전트

역할:

- `image_assets`, `showroom`, `contact` 연결 확인
- 메타데이터 흐름 점검
- 공개 페이지에서 필요한 데이터 추출 방식 검토
- 직원 응대/상담 저장 구조와 연결 확인

산출물:

- 연동 포인트 목록
- 필요한 필드 정리
- 누락 메타데이터 목록
- 기술적 리스크 목록

주의:

- 보기 좋은 구조보다 실제 운영 연결 가능성을 우선 검토

#### D. 자산 선별 에이전트

역할:

- 대표 사례 선별 기준 정리
- Before/After 우선 후보 분류
- 주력 상품 노출 현장 우선 분류
- 사진 수 부족/품질 부족 현장 보류 기준 정리

산출물:

- 대표 현장 후보 리스트
- 업종별 우선 사례
- 메인 노출군/보조 노출군/보관군 구분

주의:

- 전체 수집보다 선별 우선
- "있는 자산"보다 "쓰일 자산" 중심으로 판단

#### E. SEO/공개 배포 에이전트

역할:

- 공개 배포 구조 검토
- SEO/OG/URL 전략 검토
- CSR 유지 vs SSR 전환 판단 보조
- 검색 친화 콘텐츠 구조 제안

산출물:

- 배포 전략
- URL 구조안
- 메타 정책
- 검색/AI 검색 대응 체크리스트

주의:

- 현재 단계에서는 과한 재플랫폼 제안보다 우선순위 중심으로 판단

## 15. 파인드가구 프로젝트 기준 권장 분업

### 15-1. 현재 가장 효율적인 분업 구조

현재 단계에서 가장 추천하는 구조는 아래와 같다.

1. 전략 에이전트
2. 프론트 구현 에이전트
3. 연동 검증 에이전트
4. 자산 선별 에이전트
5. SEO/공개 배포 에이전트

대표 역할:

- 최종 우선순위 결정
- 톤 승인
- 대표 사례 승인
- 개입 기준 승인

직원 역할:

- 상담에서 자주 쓰는 설명 정리
- 자산 메타 보완

AI 역할:

- 초안 생성
- 비교안 제시
- 코드 구현
- 연동 검토
- 분류 기준 정리

### 15-2. 실제로는 무엇을 병렬로 돌릴 것인가

병렬 가능 작업:

- 홈페이지 카피 방향 정리
- 임시 프론트 시안 구현
- `image_assets` 필드 점검
- 대표 사례 선별 기준표 작성
- 공개 SEO 구조 초안 작성

순차 처리 작업:

- 최종 홈페이지 메인 구조 확정
- 자산 선별 우선순위 확정
- 라우팅/연동 반영
- 배포 구조 확정

즉, **판단은 좁히고 구현은 병렬화**하는 것이 원칙이다.

## 16. 이번 프로젝트의 AI 사용 원칙

### 16-1. 하지 말아야 할 것

- 대표가 AI 결과를 그대로 믿고 전략을 확정하는 것
- 하나의 AI 응답만 보고 브랜드 방향을 뒤집는 것
- 전체 데이터 이전을 AI가 시키는 대로 밀어붙이는 것
- 구현과 전략을 같은 세션에서 한 번에 끝내려는 것

### 16-2. 해야 할 것

- 팩트와 가설을 구분한다
- 반드시 현재 시스템과 연결 가능성을 확인한다
- 대표 개입을 줄이는 방향인지 항상 검토한다
- 직원이 실제로 쓸 수 있는 구조인지 확인한다
- 자산이 누적될수록 더 강해지는 구조인지 본다

### 16-3. 대표 판단 체크리스트

어떤 제안이 나왔을 때 아래 질문으로 거른다.

1. 이 구조는 대표 시간이 더 드는가, 덜 드는가
2. 직원이 이 흐름을 따라갈 수 있는가
3. AI가 반복 설명을 대신할 수 있는가
4. 현재 OS와 자연스럽게 연결되는가
5. 가격 비딩을 줄이는 데 도움이 되는가

위 질문에 다수 `예`가 나오는 구조를 우선 채택한다.

## 17. 지금 바로 시작할 실행 순서

### Step 1. 전략 기준 고정

- `저관여 고신뢰 구조`
- `현장 중심 + 제품 보조`
- `이중 진입 허용`
- `대표 예외 개입 원칙`

### Step 2. 자산 선별 시작

- 대표 현장 후보
- Before/After 보유 현장
- 주력 상품 노출 현장
- 메인 노출 제외 기준 적용

### Step 3. 프론트 시안 병렬 제작

- 임시 컨셉 페이지
- 이중 진입형 메인 시안
- 제품/현장 진입 차이 비교안

### Step 4. 연동 구조 점검

- `image_assets`
- `/showroom`
- `/contact`
- 상담 메타데이터 흐름

### Step 5. 공개 클라이언트 분리 검토

- 홈페이지 전용 클라이언트 필요 여부
- OS와 공유할 데이터/정책 정의
- 배포 구조 분리 검토

즉, 이 프로젝트의 실행은  
**전략 고정 -> 자산 선별 -> 프론트 시안 -> 연동 점검 -> 클라이언트 분리 검토**  
순서로 진행한다.
