# Homepage Blueprint

## 1. 문서 목적

이 문서는 파인드가구 공개 홈페이지를 현재 `FindGagu OS`와 연동하여 재구성하기 위한 실무용 설계 문서다.

핵심 방향은 다음 한 줄로 정리한다.

**이미지 자산 관리 시스템이 홈페이지 콘텐츠 엔진이 된다.**

즉, 홈페이지를 별도 외부 툴에서 수동 편집하는 구조가 아니라, 현재 OS 안의 `image_assets`와 공개 동선을 중심으로 운영 가능한 구조로 전환한다.

이번 개편의 운영 철학은 다음으로 고정한다.

**컨설팅처럼 느껴지되, 대표가 직접 많이 개입하지 않아도 돌아가는 저관여 고신뢰 구조를 만든다.**

---

## 2. 현재 확인된 팩트

### 2-1. 현재 OS 안에 이미 존재하는 공개 동선

현재 코드베이스에는 아래 공개 동선이 이미 존재한다.

- `image_assets` 기반 공개 사례 조회
- `/showroom` 공개 쇼룸 페이지
- `/contact` 공개 문의 페이지
- 사례 클릭 후 문의 맥락 전달

즉, "홈페이지 기능을 처음부터 새로 만드는 상태"는 아니다.  
공개 랜딩으로 확장 가능한 뼈대가 이미 존재한다.

### 2-2. 현재 기술 스택

- Frontend: Vite + React + TypeScript
- Router: React Router
- Backend/DB: Supabase
- 이미지: Cloudinary + Supabase Storage 이원화

### 2-3. 현재 데이터 소스

홈페이지/쇼룸의 핵심 데이터 소스 후보는 `image_assets`다.

현재 `image_assets`에서 활용 가능한 주요 필드는 다음과 같다.

- `site_name`
- `location`
- `business_type`
- `product_name`
- `color_name`
- `is_main`
- `created_at`
- `metadata.before_after_role`
- `metadata.before_after_group_id`

### 2-4. 현재 시스템의 공개 전환 구조

현재 구조상 다음 흐름은 이미 가능하다.

1. 이미지 자산 등록
2. 쇼룸에서 사례 노출
3. 사례별 맥락을 붙여 문의 페이지 진입
4. 문의 메타데이터 저장

즉, 쇼룸은 단순 갤러리가 아니라 `사례 -> 문의` 전환 퍼널의 시작점이다.

## 3. 홈페이지 재구성의 기본 판단

### 3-1. 전체 마이그레이션은 하지 않는다

기존 데이터를 "다 올리는 것"을 목표로 하지 않는다.

우선순위는 아래와 같다.

- 상담 중 바로 쓸 수 있는가
- 업종/현장/제품 태깅이 가능한가
- 대표컷이 강한가
- 현장 이해가 가능한가
- 주력 상품이 보이는가
- 리뉴얼이면 설득 포인트가 있는가
- 가능하면 Before/After가 있는가

즉, **전체 이전보다 선별 이전이 우선**이다.

### 3-2. 홈페이지의 주어는 현장으로 둔다

기존 홈페이지는 제품, 사례, 콘텐츠가 혼합된 구조였다.  
새 구조에서는 메인 서사를 다음처럼 바꾼다.

- 기존: 제품이 메인, 현장은 증거
- 새 구조: 현장이 메인, 제품은 결과를 만드는 솔루션

단, 제품 페이지나 제품 정보가 사라지는 것은 아니다.  
메인 홈페이지에서의 우선순위만 조정한다.

### 3-3. 목표는 "많이 했다"가 아니라 "진짜 잘한다"를 보여주는 것

대표 노출 현장은 많을 필요가 없다.  
중요한 것은 아래다.

- 결과물이 강한가
- 설명이 쉬운가
- 고객이 자기 공간을 상상할 수 있는가
- 실제 문의로 이어질 가능성이 높은가

따라서 사진 수가 적고 설득 포인트가 약한 현장은 메인 노출 대상에서 제외할 수 있다.

### 3-4. 컨설팅 회사 분리 전략보다 "저관여 고신뢰 구조"가 우선이다

이 문서는 별도 공간 컨설팅 회사를 새로 만드는 전략을 기본안으로 두지 않는다.

현재 더 적합한 방향은 아래다.

- 고객에게는 전문성과 기준이 느껴진다
- 홈페이지와 사례가 먼저 많은 설명을 대신한다
- 직원이 표준화된 흐름으로 1차 응대를 수행한다
- AI가 분류, 추천, 초안 작성을 보조한다
- 대표는 예외적인 고가치 건에만 제한적으로 개입한다

즉, 목표는 "대표가 더 많이 붙는 컨설팅 회사"가 아니라,  
**대표의 판단 기준이 시스템과 자산에 스며든 회사**를 만드는 것이다.

### 3-5. 개인화는 구조가 아니라 콘텐츠 수준에서 한다

고객 유형은 다르다.

- 현장/결과 중심으로 보는 고객
- 제품/규격/구성 중심으로 보는 고객
- 리뉴얼 설득이 필요한 고객
- 인테리어 업체처럼 카탈로그 성격을 먼저 확인하려는 고객

하지만 이 차이를 이유로 홈페이지의 뼈대를 방문자마다 크게 바꾸면 안 된다.  
지나치게 인터랙티브해서 페이지 구조가 계속 달라지면 고객은 적응하기 어렵다.

따라서 원칙은 아래와 같다.

- 메인 구조는 고정한다
- 고객 선택에 따라 노출 콘텐츠만 달라지게 한다
- 개인화는 섹션 내용과 추천 흐름 수준에서만 수행한다

즉, **고정 구조 + 선택형 콘텐츠**가 기본 운영 원칙이다.

## 4. 홈페이지의 역할 정의

새 홈페이지는 단순 회사 소개 페이지가 아니라 아래 6가지 역할을 수행해야 한다.

1. 업종별 대표 사례를 보여주는 공개 랜딩
2. 사례를 탐색하게 하는 쇼룸 진입점
3. 제품 카탈로그로 연결되는 중간 허브
4. 상담 전환을 만드는 퍼널
5. 직원이 설명할 내용을 먼저 대신 안내하는 1차 설명 인터페이스
6. 향후 블로그/유튜브/AI 검색에 재사용되는 콘텐츠 허브

즉, 홈페이지는 "디자인 결과물"이 아니라 "공개 콘텐츠 운영 인터페이스"이자  
**대표 시간을 절약하는 1차 설명 시스템**에 가깝다.

## 5. 공개 홈페이지 정보 구조(IA)

### 5-0. 메인 홈페이지의 운영 역할

메인 홈페이지는 진짜 컨설팅을 수행하는 곳이 아니다.  
메인 홈페이지의 역할은 아래와 같다.

- 고객이 자기 업종과 비슷한 사례를 먼저 보게 한다
- 제품 가격 비교보다 공간 결과와 기준에 먼저 집중하게 한다
- 직원이 반복 설명하던 내용을 먼저 이해시키게 한다
- 상담 전에 고객 문맥을 어느 정도 정리하게 한다
- 필요하면 자연스럽게 제품 카탈로그까지 넘어가게 한다

따라서 메인 홈페이지는 "대표 상담 유치 페이지"가 아니라,  
**비딩을 줄이고 설명 비용을 줄이는 구조적 장치**로 설계한다.

### 5-1. 메인 홈 `/`

메인 홈은 새로 만들되, 현재 OS의 공개 자산과 연동해서 구성한다.

메인 홈은 아래 퍼널 순서를 기준으로 설계한다.

1. 히어로
2. 페인포인트 선택
3. 해결 사례 제시
4. 유사 사례 둘러보기
5. 핵심 제품 브리지
6. 제품 카탈로그 진입
7. CTA 1
8. CTA 2

설명:

- `CTA 1`은 해결 사례/유사 사례를 본 직후의 상담 유도
- `CTA 2`는 제품 단계까지 본 뒤의 상담 유도

즉, 메인 홈은 단순 소개 페이지가 아니라  
**`페인포인트 -> 해결 사례 -> 유사 사례 -> 제품 -> 전환`** 흐름을 갖는 고정형 퍼널로 본다.

### 5-2. 쇼룸 `/showroom`

쇼룸은 "깊게 탐색하는 유사 사례 아카이브"로 둔다.

현재 구조를 유지하되 아래 방향으로 다듬는다.

- 홈에서 선택한 페인포인트를 이어받는다
- 유사 사례를 더 넓게 탐색하게 한다
- Before/After 또는 리뉴얼 설득 사례를 강조한다
- 사례별 문의 연결을 강화한다
- 제품 단계로 넘어가기 전 2차 설득 레이어 역할을 한다

즉, 쇼룸은 단순 갤러리가 아니라  
**`해결 사례 이후의 확장 탐색 레이어`**로 정의한다.

### 5-3. 제품 카탈로그 `/products-sites` 또는 별도 카탈로그 페이지

제품 카탈로그는 최종 구조에서 **반드시 존재해야 한다.**

이유:

- 인테리어 업체와 실무 담당자는 제품/규격/구성 중심으로 보기도 한다
- 제품 정보를 별도로 검토하려는 고객 수요가 분명히 있다
- 제품 카탈로그가 없으면 특정 고객군의 진입을 놓칠 수 있다

단, 메인 홈페이지 자체를 제품 카탈로그형으로 만들지는 않는다.

운영 원칙:

- 메인 홈에서는 제품을 사례의 맥락 안에서 먼저 소개한다
- 카탈로그는 필요한 사람이 들어갈 수 있는 별도 레이어로 둔다
- 제품은 단순 이름 나열이 아니라 `모양 + 이름 + 역할 + 적용 현장` 기준으로 보여준다
- 제품 카탈로그는 현장과 연결되어 있어야 한다

즉, 제품 카탈로그는 메인의 대체가 아니라  
**강하게 연결된 별도 탐색 레이어**다.

### 5-4. 문의 `/contact`

문의 페이지는 단순 폼이 아니라 사례 맥락을 이어받는 전환 페이지로 유지한다.

유지 원칙:

- 어떤 현장을 보고 들어왔는지 저장
- 어떤 카테고리로 유입됐는지 저장
- 어떤 상황/고민 문맥인지 저장
- 상담 카드에서 다시 활용 가능해야 함

추가 원칙:

- `CTA 1`과 `CTA 2`의 진입 맥락을 구분할 수 있어야 한다
- 빠른 상담형과 사례 기반 상담형의 문맥을 분리해서 받을 수 있어야 한다

## 6. 현재 시스템과의 연동 원칙

### 6-1. 데이터 원천

공개 홈페이지의 기본 콘텐츠 원천은 `image_assets`다.

홈페이지에 필요한 대표 데이터는 현재 필드 기준으로 먼저 구성한다.

- 현장명: `site_name`
- 지역: `location`
- 업종: `business_type`
- 제품명: `product_name`
- 색상: `color_name`
- 대표 이미지 여부: `is_main`
- Before/After: `metadata.before_after_role`, `metadata.before_after_group_id`

### 6-2. 공개 노출 기준

기본 원칙:

- 발주서, 평면도 등 내부 운영용 자료는 공개 노출 제외
- 시공 사례 사진 위주로 운영
- 상담용/세일즈용으로 실제 활용 가능한 자산 우선
- 제품 카탈로그용으로는 고객이 이해할 수 있는 대표 제품 이미지와 이름 체계를 별도로 준비할 수 있다

### 6-3. 현재 재사용할 페이지

기존 구현을 최대한 재사용한다.

- `src/pages/ShowroomPage.tsx`
- `src/pages/ContactPage.tsx`
- `src/lib/imageAssetService.ts`
- `src/App.tsx`

즉, 완전 신규 플랫폼보다는 **현재 OS 공개 동선 고도화**가 기본 전략이다.

### 6-4. AI와 자동화의 역할

AI와 자동화는 대표를 대체하는 용도가 아니라,  
대표가 반복적으로 하던 1차 설명과 분류를 줄이는 방향으로 사용한다.

우선 적용 가능한 역할:

- 업종/상황별 사례 추천 보조
- 문의 문맥 자동 세팅
- 상담 메모 초안 생성
- 직원 응대용 추천 사례/제품 묶음 보조

즉, AI는 "대표 개입 확대 장치"가 아니라  
**직원 대응 품질을 일정 수준으로 끌어올리는 보조 장치**로 본다.

## 7. 자산 선별 원칙

### 7-1. 우선 선별 대상

아래 조건을 많이 만족할수록 우선순위를 높인다.

- Before가 있다
- 결과가 잘 나왔다
- 주력 상품이 보인다
- 업종 대표성이 있다
- 리뉴얼 설득 포인트가 분명하다
- 사진 수가 최소 기준을 충족한다

### 7-2. 현장당 운영용 사진 수 기준

운영용 선별본 기준 권장 수량:

- 메인 랜딩용: 3~5장
- 쇼룸 상세용: 6~12장
- 일반 선별본: 5~8장
- 리뉴얼 설득형: Before/After 포함 4~8장

핵심은 원본 전체 보관이 아니라, **운영용 선별본을 압축해서 만드는 것**이다.

### 7-3. 대표 노출에서 제외해도 되는 경우

아래 현장은 메인 공개 노출 우선순위를 낮춘다.

- 사진 수가 너무 적다
- 대표컷이 약하다
- 주력 상품이 잘 안 보인다
- 공간 이해가 어렵다
- 리뉴얼 맥락이 부족하다
- 유사한 더 강한 현장이 이미 있다

즉, "있으니까 올린다"가 아니라 "쓸 가치가 있는지"를 기준으로 본다.

## 8. 제품과 현장의 관계 정의

새 홈페이지에서 기본 논리는 아래로 정리한다.

- 고객은 먼저 결과를 본다
- 결과를 구성하는 논리로 제품을 보여준다
- 제품은 현장을 설명하는 솔루션으로 배치한다

표현 기준:

- 메인: 현장 중심
- 쇼룸: 현장 중심 + 유사 사례 탐색
- 제품 브리지: 사례 안의 핵심 제품 설명
- 제품 카탈로그: 제품 중심

즉, 제품은 중요하지만 메인 홈페이지의 주어는 현장이다.

단, 메인 홈에서는 사용자가 처음부터 한 방향만 강요받지 않도록  
`현장부터 보기`와 `제품부터 보기`의 이중 진입 구조를 허용한다.

이때 운영 원칙은 다음과 같다.

- 현장 진입은 결과와 신뢰를 먼저 보여주는 용도
- 제품 진입은 비교보다 적용 맥락을 보여주는 용도
- 두 진입 모두 최종적으로는 사례와 상담으로 연결되어야 한다
- 제품 카탈로그는 별도 페이지로 두되, 현장과 분리된 채 독립적으로 떠다니지 않게 한다

### 8-1. 제품 단계에서 고객이 이해해야 하는 것

고객은 내부 제품명을 모를 가능성이 높다.  
따라서 제품 단계는 단순 제품명 나열로 구성하면 안 된다.

제품 단계에서는 최소한 아래 정보가 필요하다.

1. 제품 모양 또는 대표 이미지
2. 고객이 이해할 수 있는 제품명
3. 제품의 역할
4. 어떤 현장에 적용되었는지
5. 어떤 페인포인트 해결에 연결되는지

즉, 제품은 `태그`가 아니라  
**고객 언어로 번역된 설명 단위**로 보여줘야 한다.

### 8-2. 제품은 2단계로 보여준다

제품 노출은 아래 2단계 구조가 적합하다.

1. 사례 안의 핵심 제품
2. 제품 카탈로그 확장 보기

의미:

- 먼저 "이 현장을 만든 핵심 제품"을 이해하게 하고
- 그다음 "제품 전체를 더 둘러보기"로 넘어가게 한다

즉, 메인 퍼널 안에서 제품은 바로 카탈로그로 들어가는 것이 아니라  
**사례를 구성하는 핵심 제품 설명 단계**를 거친다.

## 9. AI 검색/검색 친화 구조 방향

### 9-1. 유리한 점

현재 구조는 아래 조건 때문에 향후 검색 친화 구조로 발전시킬 여지가 있다.

- 업종/지역/현장/제품 메타데이터 존재
- 이미지 alt/base text 정책 존재
- 시공 완료 후 자산을 지속적으로 추가할 수 있음

### 9-2. 아직 필요한 조건

단순히 이미지가 많다고 AI 검색에 유리해지지는 않는다.  
아래가 추가로 필요하다.

- 현장별 읽을 만한 텍스트 설명
- 업종별 랜딩 페이지
- 의미 있는 URL 구조
- 메타 태그, OG, sitemap 등 정리
- 공개 페이지 크롤링 친화성 확보

### 9-3. 렌더링 전략

현재는 Vite + React CSR 구조다.  
초기 공개는 현재 구조로도 가능하지만, 향후 검색/AI 검색 최적화가 중요해지면 아래를 검토한다.

- Next.js SSR/SSG 전환
- 또는 공개 페이지 일부만 정적/서버 렌더링 구조로 분리

즉, SSR은 지금 당장 필수는 아니지만 중장기 검토 항목이다.

## 10. 개발 단계 제안

### 1단계. 대표 자산 선별

- 대표 현장 / 업종별 대표 사례 / Before-After 우선

### 3단계. `/showroom` 고도화

- 공개 사용성 개선
- 페인포인트 진입 문맥 반영
- 대표 사례 노출 우선순위 조정
- 유사 사례 탐색 흐름 강화
- 상담 CTA 강화
- 링크 공유/진입 문맥 정리
- 직원이 바로 활용할 수 있는 사례 설명 흐름 정리

- 현재 링크 허브형 홈 제거

- 대표 직접 개입을 전제로 하지 않는 설명 구조 설계

### 5단계. 제품 브리지 및 카탈로그 정리

- 사례 안의 핵심 제품 설명 구조 정리
- 제품 대표 이미지/이름/역할 체계 정리
- 제품 카탈로그 페이지 운영 원칙 확정
- 현장과 제품의 연결 방식 정리

### 6단계. 문의 및 채널 연결 정리

- `/contact` 고도화
- `CTA 1` / `CTA 2` 유입 맥락 구분
- 채널톡/폼 연결 설계
- 유입 경로 및 상담 메타데이터 강화
- 직원 1차 대응과 대표 예외 개입 기준 분리

### 7단계. 검색 구조 보강

- 업종별 공개 페이지
- 사례 설명 텍스트 보강
- SEO/OG/크롤링 구조 개선
- 필요 시 SSR 검토

## 11. 당장 결정된 원칙

현재 시점에서 아래 원칙은 실무 기준으로 확정한다.

1. 홈페이지는 현재 OS와 연동해서 개발한다.
2. 전체 자산 이전보다 선별 이전을 우선한다.
3. 메인 홈페이지의 주어는 제품보다 현장에 둔다.
4. 메인 구조는 고정하고, 고객 선택에 따라 콘텐츠만 달라지게 한다.
5. 제품 카탈로그는 최종 구조에 반드시 포함한다.
6. 제품은 이름만이 아니라 모양, 역할, 적용 현장까지 함께 설명한다.
7. 쇼룸과 문의 동선은 현재 구현을 재사용하되, 페인포인트 기반 흐름으로 강화한다.
8. 공개 홈페이지는 장기적으로 OS 기반 콘텐츠 엔진 역할을 해야 한다.
9. 홈페이지는 대표의 직접 컨설팅을 늘리는 방향이 아니라, 설명과 분류를 먼저 수행하는 방향으로 설계한다.
10. AI와 자동화는 대표를 더 바쁘게 만드는 것이 아니라, 직원 대응 품질과 운영 효율을 높이는 용도로 사용한다.

## 13. 최종 정리

이번 홈페이지 개편은 "새 예쁜 사이트 제작"이 아니다.

이번 작업의 본질은 아래다.

- 시공 이미지 자산을 운영 가능한 구조로 선별하고
- 그 자산을 현재 OS 안에서 메인 홈, 쇼룸, 제품 카탈로그, 문의 동선과 연결하고
- 쇼룸, 카탈로그, 문의, 상담, 향후 블로그/유튜브/검색까지 이어지는
- 하나의 콘텐츠 운영 시스템으로 만드는 것
- 그리고 대표가 반복 설명하던 기준을 홈페이지, 자산, 직원 응대, AI 보조 흐름에 녹여 넣는 것
- 동시에 고객마다 홈페이지가 낯설게 바뀌지 않도록 고정된 구조 안에서 콘텐츠만 선택적으로 바꾸는 것

따라서 홈페이지 개발은 디자인 작업이 아니라,  
**현재 시스템과 연결된 공개 콘텐츠 엔진 구축 작업**이며,  
동시에 **저관여 고신뢰 운영 구조를 만드는 작업**으로 정의한다.

## 14. AI 오케스트레이션 운영안

### 14-1. 왜 오케스트레이션이 필요한가

홈페이지 개편은 단순 프론트 작업이 아니다.  
현재 과업은 아래 작업이 동시에 얽혀 있다.

- 브랜드 포지셔닝 정리
- 홈페이지 구조 설계
- 현재 OS와의 연동
- 공개 배포/검색 구조 검토
- 이미지 자산 선별 및 메타 정리

이 모든 작업을 하나의 AI나 하나의 세션에서 처리하면 품질이 흔들리기 쉽다.  
따라서 이 프로젝트는 **역할별 AI 분업과 오케스트레이션**을 기본 운영 방식으로 둔다.

핵심 원칙:

- 전략과 구현을 분리한다
- 공개 홈페이지와 내부 OS 관점을 분리한다
- 자산 선별과 프론트 구현을 분리한다
- 최종 의사결정은 대표가 하되, 정보 수집과 초안 생성은 AI가 분담한다

### 14-2. 기본 역할 분리

#### A. 전략 에이전트

역할:

- 홈페이지 방향과 철학 정리
- 제품 중심 vs 현장 중심 판단
- 카피 톤과 진입 구조 검토
- 대표 개입 최소화 원칙 점검

산출물:

- 구조안
- 카피 방향
- 우선순위 문서
- 의사결정 기준표

주의:

- 직접 구현보다 판단 기준 정리에 집중
- 감각적 제안과 확정 사실을 반드시 구분

#### B. 프론트 구현 에이전트

역할:

- 실제 페이지 구현
- 컴포넌트 구성
- 반응형 레이아웃 정리
- 공개용 페이지 품질 개선

산출물:

- 페이지 컴포넌트
- 라우팅 연결
- UI 수정안
- 실제 동작 가능한 프로토타입

주의:

- OS 내부 로직을 함부로 건드리지 말 것
- 공개 홈페이지용 코드와 내부 운영 코드를 혼합하지 말 것

#### C. 연동 검증 에이전트

역할:

- `image_assets`, `showroom`, `contact` 연결 확인
- 메타데이터 흐름 점검
- 공개 페이지에서 필요한 데이터 추출 방식 검토
- 직원 응대/상담 저장 구조와 연결 확인

산출물:

- 연동 포인트 목록
- 필요한 필드 정리
- 누락 메타데이터 목록
- 기술적 리스크 목록

주의:

- 보기 좋은 구조보다 실제 운영 연결 가능성을 우선 검토

#### D. 자산 선별 에이전트

역할:

- 대표 사례 선별 기준 정리
- Before/After 우선 후보 분류
- 주력 상품 노출 현장 우선 분류
- 사진 수 부족/품질 부족 현장 보류 기준 정리

산출물:

- 대표 현장 후보 리스트
- 업종별 우선 사례
- 메인 노출군/보조 노출군/보관군 구분

주의:

- 전체 수집보다 선별 우선
- "있는 자산"보다 "쓰일 자산" 중심으로 판단

#### E. SEO/공개 배포 에이전트

역할:

- 공개 배포 구조 검토
- SEO/OG/URL 전략 검토
- CSR 유지 vs SSR 전환 판단 보조
- 검색 친화 콘텐츠 구조 제안

산출물:

- 배포 전략
- URL 구조안
- 메타 정책
- 검색/AI 검색 대응 체크리스트

주의:

- 현재 단계에서는 과한 재플랫폼 제안보다 우선순위 중심으로 판단

- 대표가 AI 결과를 그대로 믿고 전략을 확정하는 것
- 하나의 AI 응답만 보고 브랜드 방향을 뒤집는 것
- 전체 데이터 이전을 AI가 시키는 대로 밀어붙이는 것
- 구현과 전략을 같은 세션에서 한 번에 끝내려는 것

### 16-2. 해야 할 것

- 팩트와 가설을 구분한다
- 반드시 현재 시스템과 연결 가능성을 확인한다
- 대표 개입을 줄이는 방향인지 항상 검토한다
- 직원이 실제로 쓸 수 있는 구조인지 확인한다
- 자산이 누적될수록 더 강해지는 구조인지 본다

### 16-3. 대표 판단 체크리스트

어떤 제안이 나왔을 때 아래 질문으로 거른다.

1. 이 구조는 대표 시간이 더 드는가, 덜 드는가
2. 직원이 이 흐름을 따라갈 수 있는가
3. AI가 반복 설명을 대신할 수 있는가
4. 현재 OS와 자연스럽게 연결되는가
5. 가격 비딩을 줄이는 데 도움이 되는가

위 질문에 다수 `예`가 나오는 구조를 우선 채택한다.

## 17. 지금 바로 시작할 실행 순서

### Step 1. 전략 기준 고정

- `저관여 고신뢰 구조`
- `현장 중심 + 제품 보조`
- `고정 구조 + 선택형 콘텐츠`
- `페인포인트 기반 퍼널`
- `제품 카탈로그 필수`
- `대표 예외 개입 원칙`

### Step 2. 자산 선별 시작

- 대표 현장 후보
- Before/After 보유 현장
- 주력 상품 노출 현장
- 메인 노출 제외 기준 적용

### Step 3. 프론트 시안 병렬 제작

- 임시 컨셉 페이지
- 메인 퍼널 시안
- 쇼룸 확장 탐색 시안
- 제품 브리지/카탈로그 시안

### Step 4. 연동 구조 점검

- `image_assets`
- `/showroom`
- `/products-sites`
- `/contact`
- 상담 메타데이터 흐름

### Step 5. 공개 클라이언트 분리 검토

- 홈페이지 전용 클라이언트 필요 여부
- OS와 공유할 데이터/정책 정의
- 배포 구조 분리 검토

즉, 이 프로젝트의 실행은  
**전략 고정 -> 자산 선별 -> 프론트 시안 -> 연동 점검 -> 클라이언트 분리 검토**  
순서로 진행하되, 메인/쇼룸/카탈로그 3단 구조를 유지한 채 세부 콘텐츠만 발전시키는 방식으로 진행한다.

## 18. 총괄 역할 분리 확정안

### 18-1. 총괄

총괄은 본 대화의 메인 에이전트가 맡는다.

총괄 역할:

- 전체 우선순위 결정
- 작업 단위 분해
- 역할 간 충돌 방지
- OS와 홈페이지 경계 관리
- 팩트/가설 구분
- 대표에게 최종 판단 포인트만 보고

총괄은 직접 모든 구현을 하지 않는다.  
대신 어떤 역할이 무엇을 먼저 해야 하는지 정하고, 결과물을 검토하고, 다음 작업을 배치한다.

### 18-2. OS 에이전트

OS 에이전트의 책임 범위:

- 데이터 마이그레이션
- `image_assets` 품질 보정
- 자산 메타데이터 정리
- 관리자용 자산 관리 화면
- 상담/백오피스 로직 안정화
- 공개 홈페이지에 제공할 데이터 계약 유지

OS 에이전트가 건드려도 되는 것:

- 자산 스키마 및 메타 기준
- 관리자 입력 흐름
- 내부 운영 페이지
- 상담 저장/조회 로직

OS 에이전트가 건드리면 안 되는 것:

- 공개 홈페이지 카피 방향
- 쇼룸의 공개 UX 우선순위
- 메인 랜딩 구조
- 공개 CTA 문구

한 줄 정의:

**OS 에이전트는 엔진과 운영을 맡는다.**

### 18-3. 홈페이지 에이전트

홈페이지 에이전트의 책임 범위:

- 메인 홈
- `/showroom`
- `/products-sites` 또는 제품 카탈로그 페이지
- `/contact`
- 공개 탐색 동선
- 공개 카피와 전환 구조
- SEO/OG/공개 URL 구조

홈페이지 에이전트가 건드려도 되는 것:

- 쇼룸 카드 배치
- 공개 랜딩 섹션 구조
- 현장/제품 진입 UX
- 제품 브리지 설명 구조
- 제품 카탈로그 정보 구조
- CTA 구조
- 공개 페이지 톤

홈페이지 에이전트가 건드리면 안 되는 것:

- `image_assets` 의미 자체 변경
- 관리자 메타 입력 기준 임의 변경
- 상담 DB 구조 임의 변경
- OS 내부 화면 로직 변경

한 줄 정의:

**홈페이지 에이전트는 보여주는 층과 전환을 맡는다.**

### 18-4. 공유 계약 담당

OS와 홈페이지 사이에는 반드시 공유 계약이 있어야 한다.

공유 계약 범위:

- 공개 노출 가능한 필드 목록
- 대표 이미지 기준
- Before/After 기준
- 업종/제품/색상 태그 기준
- 공개 제외 자산 기준
- 상담으로 넘길 때 필요한 메타데이터 목록

이 계약 변경은 총괄 승인 후에만 한다.

한 줄 정의:

**공유 계약은 두 에이전트가 같이 쓰는 약속이다.**

### 18-5. 검증 담당

검증 담당은 총괄이 직접 하거나, 필요 시 별도 검토 역할로 둔다.

검증 항목:

- 공개 페이지 수정이 OS를 깨지 않는지
- OS 메타 변경이 공개 페이지를 깨지 않는지
- 데이터 계약과 실제 구현이 맞는지
- 대표 개입을 늘리는 방향으로 흐르지 않는지

한 줄 정의:

**검증은 품질보다 경계 확인이 먼저다.**

## 19. 실제 운영 규칙

### 19-1. 승인 없이 건드리면 안 되는 것

- `image_assets` 공개 기준 변경
- 상담 메타데이터 저장 구조 변경
- 공개/내부 공용 유틸의 의미 변경
- 라우팅 대규모 재구성
- OS와 홈페이지 공통 컴포넌트의 전역 스타일 변경

### 19-2. 홈페이지 에이전트가 새 요구를 내는 방식

홈페이지 에이전트는 아래 형식으로만 요청한다.

- 왜 필요한가
- 어떤 공개 UX를 위해 필요한가
- 어떤 필드가 더 필요한가
- 기존 OS를 깨지 않고 가능한가

예:

- 홈페이지용 `featured` 플래그가 필요함
- 리뉴얼 랜딩용 `before_after_priority`가 필요함
- 업종 진입 카피에 쓸 짧은 설명 필드가 필요함

### 19-3. OS 에이전트가 반영할 때 기준

OS 에이전트는 아래를 먼저 본다.

- 관리자 입력 부담이 커지지 않는가
- 기존 마이그레이션에 치명적이지 않은가
- 공개 홈페이지 외에도 재사용 가능한가
- 장기적으로 데이터 품질이 좋아지는가

## 20. 지금부터의 실제 배치

### 20-1. 총괄 기준 현재 배치

현재 배치는 다음으로 고정한다.

1. 총괄: 메인 에이전트
2. OS 에이전트: 마이그레이션 + 자산 관리 + 데이터 계약
3. 홈페이지 에이전트: 메인 홈 + 쇼룸 + 문의 + 공개 전환

### 20-2. 당장 홈페이지 에이전트에 넘길 일

- `/homepage-concept` 2차 수정
- `현장부터 보기 / 제품부터 보기` 이중 진입 구조
- 쇼룸을 홈페이지 관점으로 재정의
- 공개 CTA 흐름 다듬기
- 메인 홈과 쇼룸의 관계 정리

### 20-3. 당장 OS 에이전트에 남길 일

- 이미지 자산 마이그레이션
- 메타데이터 보정
- 공개에 필요한 대표 자산 선별 지원
- 관리자 화면 안정화

### 20-4. 총괄이 계속 체크할 것

- 쇼룸이 홈페이지 소관으로 이동하면서 OS를 건드리지 않는지
- 홈페이지 쪽 요구가 실제 운영 부담을 늘리지 않는지
- 대표 개입 최소화 원칙이 유지되는지
- 지금 만드는 구조가 장기적으로 클라이언트 분리에도 유리한지
