# 콘텐츠 시스템 설계도

## 1. 목적
- 이미지 자산과 쇼룸에서 출발한 사례 데이터를 `발행 가능한 콘텐츠 자산`으로 바꾼다.
- 내부 운영자는 한 곳에서 `기획 -> 원문 -> 파생 -> 자동화 -> 배포 반영`까지 끝낼 수 있어야 한다.
- 외부 채널은 블로그가 메인 스토리, 카드뉴스와 쇼츠는 후킹, 롱폼은 확장 설명 역할을 맡는다.

## 2. 핵심 화면
- `발행 큐`: 오늘 처리할 콘텐츠를 고르고 우선순위를 정하는 화면
- `콘텐츠 상세`: 하나의 콘텐츠를 실제로 만드는 화면
- `배포 관리`: 채널별 상태와 URL을 정리하는 화면
- `자동화 큐`: 요청 상태, payload, 재요청, 디스패치를 관리하는 화면
- `템플릿`: 반복 가능한 원문/파생 구조를 관리하는 화면

## 3. 핵심 흐름
1. 쇼룸 또는 자산 기반으로 콘텐츠 후보를 만든다.
2. 발행 큐에서 오늘 처리 대상을 고른다.
3. 콘텐츠 상세에서 원문과 파생 초안을 정리한다.
4. 채널별 payload를 확인하고 자동화 요청을 만든다.
5. 자동화 큐에서 처리 상태를 추적한다.
6. 배포 관리에서 URL과 최종 상태를 반영한다.

## 4. 운영 기준
- 새로고침은 전체 페이지 재진입이 아니라 화면 내 부분 재조회로 해결한다.
- 운영자는 항상 `무엇이 최신인지`, `무엇이 막혔는지`, `무엇을 먼저 눌러야 하는지`를 화면 안에서 이해할 수 있어야 한다.
- 템플릿은 편집 편의보다 운영 일관성을 위해 존재한다.
- `연동 미설정` 상태에서는 자동화 요청 생성을 막는 것이 기본 원칙이다.

## 5. 데이터 원칙
- 콘텐츠는 단일 채널용 초안이 아니라 여러 파생 채널의 중심 원본으로 취급한다.
- 블로그 원문은 SEO/AEO 기준의 메인 본문이다.
- 카드뉴스, 쇼츠, 롱폼, SNS 캡션은 블로그 원문에서 파생되되 동일한 메시지 계층을 유지해야 한다.
- 공개 수준은 `teaser`, `summary`, `detail`처럼 정보 공개량을 조절하는 개념으로 해석한다.
