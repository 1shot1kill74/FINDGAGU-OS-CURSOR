# 콘텐츠 시스템 운영 절차

## 1. 목적
- 이 문서는 내부 운영자가 콘텐츠를 기획하고 발행 준비를 확인하고 자동화 요청을 생성한 뒤 최종 배포 반영까지 한 바퀴 돌리는 기준 절차를 정의한다.

## 2. 시작 전 전제
- 내부 운영용 Google 계정으로 로그인된 브라우저 세션이 준비되어 있어야 한다.
- `발행 큐`, `콘텐츠 상세`, `배포 관리`, `자동화 큐`, `템플릿` 5개 화면은 모두 같은 콘텐츠 기준으로 연결되어야 한다.
- 페이지 재진입 없이도 각 화면의 새로고침 버튼으로 최신 상태를 다시 확인할 수 있어야 한다.
- 개발 단계에서는 `VITE_CONTENT_WORKSPACE_SOURCE=auto | local | supabase` 기준으로 데이터 원천이 달라질 수 있으며, 화면은 현재 원천 라벨을 보여줘야 한다.
- 저장/상태 변경 액션은 가능하면 `Supabase 원천`에 먼저 반영을 시도하고, 실패하면 `로컬 워크스페이스`에 fallback 되어야 한다.
- 각 화면은 마지막 저장 결과가 `Supabase`, `로컬`, `로컬 fallback` 중 어디에 반영되었는지 보여줘야 한다.

## 3. 기본 운영 순서
1. `발행 큐`에서 오늘 처리할 콘텐츠를 고른다.
2. `콘텐츠 상세`에서 기본 정보, 블로그 원문, 파생 콘텐츠, 배포 상태를 정리한다.
3. `자동화 Payload 미리보기`에서 현재 채널과 작업유형이 맞는지 확인한다.
4. `자동화 요청 생성`으로 큐에 적재한다.
5. `자동화 큐`에서 상태, payload, 최근 오류, 반영 메타를 확인한다.
6. `배포 관리`에서 채널별 최종 상태와 발행 URL을 확인한다.

## 4. 발행 큐에서 보는 순서
- 쇼룸에서 새 이미지/현장이 들어왔으면 먼저 `쇼룸에서 동기화`를 실행해 콘텐츠 후보를 갱신한다.
- `운영 진단`에서 critical/warning 신호가 먼저 있는지 보고, 있으면 해당 링크로 바로 들어간다.
- 외부 연동 직전에는 `실운영 최종 점검`에서 pass/attention 현황을 먼저 보고, attention 항목 링크로 바로 이동한다.
- `연동 보완` 신호가 있는지 먼저 본다.
- `목록 새로고침` 후 `마지막 목록 갱신`이 현재 시점으로 바뀌는지 본다.
- 필요하면 `연동 상태 새로고침`으로 웹훅 준비 상태를 최신 스냅샷 기준으로 다시 확인한다. 이때 권장 힌트는 고정 문구가 아니라 실제 시각 비교 결과를 따라야 한다.
- `오늘 할 일 3줄`과 `우선 분류`를 보고 오늘 처리 순서를 정한다.
- 목록에서 콘텐츠를 하나 선택한 뒤 `우선 처리 근거`, `준비 체크`, `막힘 신호`, `최근 활동`을 함께 보고 바로 상세/배포/자동화 화면으로 이동한다.
- 원천 추적이 필요하면 `원천 연결 요약` 또는 `원천 자산 샘플`에서 바로 `이미지 자산 관리`로 이동해 해당 자산 카드에 포커스한다.
- 필요하면 `공유용 요약`을 복사해 운영 메시지 초안으로 바로 사용한다.
- 쇼룸 동기화로 새 후보가 들어오면 기본 배포 채널과 기본 파생초안이 함께 준비되어야 한다.
- `Supabase` 사용 중이면 쇼룸 동기화 결과는 콘텐츠/원문/배포/파생초안/활동 로그까지 함께 원격 반영을 시도해야 한다.

## 5. 콘텐츠 상세에서 보는 순서
- `콘텐츠 준비도`, `자동화 준비도`, `준비 체크리스트`를 본다.
- 선택한 채널 기준 `웹훅 준비`가 `연동 미설정 / mock 연결 / 실URL 연결` 중 무엇인지 확인한다.
- `마지막 콘텐츠 갱신`, `연동 상태 확인`, `마지막 작업 목록 갱신` 시각을 각각 구분해서 본다.
- 세 시각이 크게 어긋나면 화면이 안내하는 새로고침 권장 힌트에 따라 `콘텐츠 새로고침`, `연동 상태 새로고침`, `작업 새로고침` 중 필요한 버튼을 누른다. 각 버튼은 페이지 재진입 없이 최신 스냅샷을 다시 읽어와야 한다.
- `파생 콘텐츠` 탭에서 채널별 초안 상태와 후킹/개요를 확인한다.
- `원천 연결` 섹션에서는 쇼룸 그룹 키, `space id`, 연결 자산 수를 보고, 샘플 자산을 눌러 `이미지 자산 관리` 화면으로 바로 이동할 수 있어야 한다.
- `배포 상태` 탭에서는 채널을 선택한 뒤 `발행 준비 체크`, `채널 파생 초안`, `최근 자동화 작업`, `최근 활동 로그`를 함께 보고 지금 상태가 왜 만들어졌는지 해석한다.
- `배포 상태` 탭의 `발행 완료` 액션은 이미 완료된 채널이나 `연동 미설정` 채널에는 직접 허용하지 않는다.

## 6. 자동화 요청 생성 기준
- 블로그 원문 또는 파생 초안이 최소 기준 이상 채워져 있어야 한다.
- 채널별 필수 메타가 비어 있지 않아야 한다.
- 선택한 작업유형의 웹훅이 최소 `mock 연결` 이상이어야 한다.
- 운영 시작 전 실 검증에서는 `실URL 연결` 상태를 권장한다.
- 원문 저장 시 블로그 원문뿐 아니라 연결된 기본 파생초안과 활동 로그도 함께 최신 상태로 유지되어야 한다.

## 7. 자동화 큐에서 보는 순서
- `연동 준비 상태` 배너부터 본다.
- 상단 `실운영 준비도`가 `운영 시작 가능 / 부분 준비 / mock 단계 / 미설정` 중 어디인지 먼저 확인한다.
- 채널별 준비 카드를 보고 어떤 채널이 실URL인지, 어떤 채널이 아직 mock/미설정인지 우선순위를 정한다.
- `확인 시각`과 `마지막 목록 갱신`을 함께 본다.
- `새로고침 권장 힌트`가 보이면 `상태 새로고침` 또는 `새로고침` 중 필요한 버튼을 먼저 누른다.
- 요청 상세에서는 `전송 준비 체크`를 먼저 보고, 콘텐츠 제목/SEO/CTA/FAQ/채널 배포 레코드/채널 파생 초안이 준비되었는지 확인한다.
- `Payload 미리보기`는 최소 `작업 메타`, `콘텐츠 기본 정보`, `채널 배포 정보`, `파생 초안`, `최근 활동 맥락`, `준비 체크 결과`를 포함해야 한다.
- `웹훅 호출`은 브라우저가 외부 URL을 직접 호출하지 않고 `content-automation-dispatch` Edge Function을 통해 서버사이드 secret과 채널별 URL을 사용해야 한다.
- 상태 전이는 기본적으로 `queued -> processing -> completed` 순서를 따르고, `failed` 항목은 직접 완료 처리하지 않고 `재요청`으로 새 작업을 생성한다.
- 자동화 상태가 바뀌면 연결된 배포 채널 상태와 콘텐츠 상태도 함께 자동 반영되어야 하며, 화면은 이 결과를 한 줄 요약으로 보여줘야 한다.
- 실패 항목은 최근 오류와 반영 메타를 함께 보고, 재요청 전에 payload와 웹훅 준비 상태를 다시 확인한다.

## 8. 배포 관리에서 보는 순서
- 채널별 `연동 미설정`, `생성 불가`, `확인 필요` 신호를 먼저 본다.
- `목록 새로고침` 후 `마지막 목록 갱신`이 바뀌는지 확인한다.
- 외부 설정을 바꿨다면 `연동 상태 새로고침`으로 최신 스냅샷 재조회 후 판정이 즉시 바뀌는지 확인한다.
- 목록에서 채널을 하나 선택한 뒤, 우측 상세에서 `발행 준비 체크`와 `최근 자동화 작업`, `최근 활동`, `채널 파생 초안`을 함께 본다.
- `발행 완료` 처리는 이미 완료된 항목이나 `연동 미설정` 항목에는 직접 허용하지 않는다.
- 채널 상태가 바뀌면 콘텐츠 상태와 연동 준비 상태도 함께 자동 반영되어야 하며, 화면은 이 결과를 한 줄 요약으로 보여줘야 한다.
- 발행 URL이 들어온 뒤에는 콘텐츠 상세와 자동화 큐의 상태와 서로 모순되지 않는지 교차 확인한다.

## 9. 실운영 1회전 검증 흐름
1. 자동화 큐에서 대상 작업유형이 `실URL 연결`인지 확인한다.
2. 발행 큐에서 `목록 새로고침`과 `연동 상태 새로고침`을 각각 1회 실행한다.
3. 발행 큐의 `실운영 최종 점검`에서 attention 항목이 무엇인지 먼저 확인한다.
4. 콘텐츠 상세에서 `콘텐츠 새로고침`과 `연동 상태 새로고침`을 각각 1회 실행한다.
5. `자동화 Payload 미리보기`와 `자동화 요청 생성` 가능 상태를 확인한다.
6. 요청을 생성한 뒤 `최근 자동화 작업`에 즉시 반영되는지 본다.
7. 자동화 큐에서 상태 변화를 확인한다.
8. 배포 관리에서 최종 채널 상태와 발행 URL 반영을 확인한다.

## 10. 통과 기준
- 각 화면의 새로고침 버튼이 페이지 재진입 없이 최신 상태를 반영해야 한다.
- `마지막 ... 갱신` 문구가 버튼 실행 직후 현재 시점으로 바뀌어야 한다.
- `연동 상태 확인` 문구가 비어 있지 않고, 확인 중 또는 실패 또는 확인 시각 중 하나를 명확히 보여줘야 한다.
- 최소 1개 채널은 `발행 큐 -> 콘텐츠 상세 -> 자동화 큐 -> 배포 관리` 흐름이 끊김 없이 이어져야 한다.
- 운영 패널의 막힘 신호나 추천 후보를 눌렀을 때 관련 화면/탭/채널로 바로 이동할 수 있어야 한다.
- 운영 진단은 최소 `상태 불일치`, `발행 URL 누락`, `실패 사유 누락`, `배포/파생 누락` 신호를 잡아낼 수 있어야 한다.
- 쇼룸 원천 데이터에서 동기화한 신규 현장이 발행 큐와 배포 관리에 모두 보이는지 확인되어야 한다.
- 콘텐츠 저장, 자동화 요청, 배포 상태 변경 같은 주요 조작은 최근 활동 로그에 남아야 한다.
- 자동화 작업 상태와 배포 상태가 서로 어긋나지 않도록 `자동화 -> 배포 -> 콘텐츠` 방향의 상태 자동 반영이 유지되어야 한다.
- `Supabase` 저장이 실패한 경우에도 사용자 작업은 사라지지 않고 로컬 fallback 상태로 남아야 하며, 화면은 이를 토스트로 알려줘야 한다.
- 활동 로그 원격 반영은 중복 insert로 터지지 않도록 id 기준 `upsert` 성격으로 처리되어야 한다.

## 11. 템플릿에서 보는 순서
- 템플릿 목록에서 운영 상태와 사용 횟수를 먼저 본다.
- 상세 패널에서는 `운영 가이드`, `운영 체크`, `추천 적용 후보`를 먼저 보고 실제 저장 여부를 결정한다.
- `안정적` 템플릿은 기본값 후보로, `테스트 확대`와 `신규`는 제한 적용 대상으로 본다.
- `보완 필요` 또는 `비활성` 상태는 설명/구조를 먼저 다듬은 뒤 다시 저장한다.
