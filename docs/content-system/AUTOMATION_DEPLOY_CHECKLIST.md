# 자동화 배포 체크리스트

## 1. 배포 전 기본 확인
- 내부 운영용 Google 계정으로 로그인된 브라우저 세션이 준비되어 있다.
- 대상 Supabase Edge Function이 배포되어 있다.
  - 내부 호출용 `content-automation-dispatch`
  - 외부 callback용 `content-automation-callback`
- 웹훅 URL과 secret 값이 환경변수에 설정되어 있다.
- mock URL과 실URL을 혼용하지 않을 기준이 팀 내에서 정해져 있다.
- `docs/content-system/WEBHOOK_CONTRACT.md` 기준 payload와 응답 형식이 외부 자동화와 합의되어 있다.

## 2. 준비 상태 확인
- 자동화 큐 `연동 준비 상태`에서 각 작업유형이 `미설정 / mock 단계 / 부분 준비 / 운영 시작 가능` 중 무엇인지 확인한다.
- 최소 1개 운영 채널은 `실URL 연결` 상태여야 한다.
- `확인 시각`이 현재 시점과 크게 벌어져 있지 않아야 한다.

## 3. 화면별 새로고침 검증
### 발행 큐
- `목록 새로고침` 후 `마지막 목록 갱신`이 바뀐다.
- `연동 상태 새로고침` 후 `연동 상태 확인`이 바뀐다.

### 콘텐츠 상세
- `콘텐츠 새로고침` 후 `마지막 콘텐츠 갱신`이 바뀐다.
- `연동 상태 새로고침` 후 `연동 상태 확인`이 바뀐다.
- `작업 새로고침` 후 `마지막 작업 목록 갱신`이 바뀐다.

### 자동화 큐
- `새로고침` 후 `마지막 목록 갱신`이 바뀐다.
- `상태 새로고침` 후 `확인 시각`이 바뀐다.

### 배포 관리
- `목록 새로고침` 후 `마지막 목록 갱신`이 바뀐다.
- `연동 상태 새로고침` 후 연동 판정이 즉시 다시 계산된다.

## 4. 1차 내부 테스트
1. 발행 큐에서 테스트 콘텐츠를 하나 고른다.
2. `운영 진단`에 critical 신호가 없는지 먼저 본다.
3. 콘텐츠 상세에서 기본 정보와 블로그/파생 초안이 준비됐는지 확인한다.
4. 웹훅 준비 상태가 최소 `mock 연결` 이상인지 확인한다.
5. 자동화 요청을 생성한다.
6. 자동화 큐에서 payload, 상태, 최근 오류, `상태 자동 반영 결과`를 확인한다.
7. 필요하면 `웹훅 호출` 또는 `실패 항목 일괄 재요청`을 실행한다.
8. 배포 관리에서 상태와 URL 반영, `상태 자동 반영 결과`를 확인한다.

## 5. 실URL 1회전 검증
1. 자동화 큐에서 대상 작업유형이 `실URL 연결`인지 확인한다.
2. 발행 큐와 콘텐츠 상세에서 각각 새로고침을 1회씩 수행한다.
3. 자동화 요청 생성 후 자동화 큐에서 실제 반영을 확인한다.
4. 외부 자동화 완료 뒤 배포 관리에서 최종 URL을 확인한다.
5. 다시 콘텐츠 상세로 돌아와 상태, URL, 최근 작업이 서로 모순되지 않는지 본다.

## 6. 완료 기준
- 새로고침 버튼이 페이지 재진입 없이 최신 상태를 반영한다.
- `마지막 ... 갱신` 문구가 실제 현재 시각과 크게 어긋나지 않는다.
- 최소 1개 채널은 요청 생성부터 배포 URL 반영까지 끊김 없이 이어진다.
- 차단 신호가 남아 있다면 운영 시작 완료로 판단하지 않는다.
