# 프로젝트 컨텍스트 (CONTEXT)

> 이 문서는 FINDGAGU OS의 최상위 운영 원칙을 담는 `헌법 문서`입니다.  
> 구현 상세는 `BLUEPRINT.md`, 과거 경위는 `docs/history/context-history.md`와 `docs/history/journal-2026-q1.md`를 참조합니다.

## 1. 프로젝트 정의

- 서비스명: `FINDGAGU OS`
- 성격: 가구/인테리어 실무의 상담, 견적, 시공 사례, 후속 상담을 하나의 운영 흐름으로 묶는 통합 시스템
- 핵심 가치: `현장 경험과 시공 사례를 디지털 자산으로 바꾸는 것`
- 현재 공개 전략: 고객 공개 경험의 중심은 `쇼룸형 홈페이지`

## 2. 현재 방향

- 1순위는 `이미지 자산 허브`를 안정적으로 만드는 것입니다.
- 쇼룸은 단순 갤러리가 아니라, 고객이 사례를 탐색하고 스스로 관심 현장을 좁혀 가는 설득 구조여야 합니다.
- `쇼룸형 홈페이지`는 메인 공개 경험 정의를 유지하지만, 당분간 구현 작업은 보류 상태로 둡니다.
- 채널톡은 메인 쇼룸의 대체가 아니라 `후속 정보 제공 + 행동 촉구` 레이어입니다.
- 공개 동선은 먼저 열 수 있어도, 내부 운영 화면은 반드시 분리된 보호 구간으로 유지합니다.

## 3. 운영 원칙

### 정보 계층화

- 상담 타임라인은 흐름과 판단에 집중합니다.
- 실측 PDF, 대량 이미지, 견적 이력 전체 같은 기술 자산은 전용 모듈에서 다룹니다.
- 한 화면에 모든 정보를 쌓아 넣지 않고 역할별 진입점으로 분리합니다.

### 데이터 무결성

- 동일 고객의 정보와 등급은 전역적으로 일관되어야 합니다.
- 상담 상태는 실무자의 판단값이며, 시스템은 판단을 대체하지 않습니다.
- 자동화는 `최근 활동`, `견적 흔적`, `관심 사례` 같은 보조 신호를 제공하는 쪽에 우선합니다.

### 단계적 론칭

- 백오피스 전체 완성을 기다리지 않고 공개 동선과 내부 운영 동선을 분리해 론칭합니다.
- 1차 공개 범위는 `쇼룸`, `문의`, `채널톡 연결` 같은 고객-facing 흐름입니다.
- 운영 화면과 관리자 기능은 인증 뒤에서만 엽니다.

### 사례 중심 해석

- 이 시스템의 중심 자산은 문서가 아니라 `사례`입니다.
- 시공 사진, 전후 비교, 현장 설명, 예산 밴드, 해결 포인트는 모두 하나의 사례 문맥으로 연결되어야 합니다.
- 고객은 링크보다 `내 상황과 비슷한 사례의 해석`에 더 큰 가치를 느낍니다.

## 4. 사용자 관점

### 대표/관리자

- 전체 운영 기준, 우선순위, 자동화 방향을 결정합니다.
- 결과보다 `실무 효율`과 `데이터 자산화`를 중요하게 봅니다.

### 상담 직원

- 상담 관리, 견적 발행, 고객 케어가 핵심 역할입니다.
- 필요한 것은 화려한 화면보다 `현재 문맥을 빨리 이해하고 바로 행동할 수 있는 구조`입니다.

### 현장 실측/시공팀

- 실측 자료, 현장 정보, 시공 사진을 다룹니다.
- 타임라인보다는 전용 입력/열람 모듈이 더 중요합니다.

### 고객

- 단순히 사진을 보는 것이 아니라, `비슷한 사례`, `공간 변화`, `다음 단계`를 이해하려고 들어옵니다.
- 메인 경험은 `쇼룸형 홈페이지`에서 끝나야 하며, `오픈쇼룸`은 보조 공개 수단, 채널톡은 그 뒤를 잇는 후속 레이어여야 합니다.

## 5. 핵심 도메인 해석

### 쇼룸

- `쇼룸형 홈페이지`는 고객 공개 메인 경험으로 정의된 별도 표면입니다.
- `내부쇼룸`은 `/showroom`에서 여는 내부 설명/공유용 쇼룸입니다.
- `오픈쇼룸`은 `/public/showroom`에서 여는 외부 공개 쇼룸입니다.
- 로컬 개발에서는 로그인 흐름이 배포된 Vercel 주소로 이동하는 문제가 있어, `내부쇼룸` 수정 시 로그인 대신 현재 우회 진입 경로를 사용합니다.
- 이 로그인/리다이렉트 문제가 정리되기 전까지 `내부쇼룸` 로컬 수정은 우회 진입 기준으로 작업합니다.
- `Before/After`는 일반 사례와 성격이 다르므로 별도 문맥에서 보여줍니다.
- 쇼룸 문의에는 `showroom_context`, `showroom_entry_label` 같은 유입 문맥이 남아야 합니다.

### 채널톡

- 초기에는 AI 대화형보다 `폼 기반 정보 수집`이 우선입니다.
- 수집 순서는 `도움 유형 -> 업종 -> 면적 -> 지역 -> 전화번호`입니다.
- 쇼룸 링크는 정보를 먼저 확보한 뒤 보조적으로 제안합니다.

### 상담 관리

- 상담 관리는 다른 화면과 분리된 `작업형 대시보드`입니다.
- 선택 유지, 상태 변경 후 페이지 추적, 우측 상세 패널 기준판 유지가 중요합니다.
- 상담 상태는 `접수 / 견적 / 진행 / 완료 / AS / 무효 / 거절` 7종을 기준으로 봅니다.

### 이미지 자산

- 이미지 자산 관리는 쇼룸, 상담 보조, 콘텐츠 자동화의 공통 허브입니다.
- 운영 목적은 단순 보관이 아니라 `찾기`, `분류`, `공유`, `재활용`입니다.

## 6. 기술/운영 전제

- 개발 환경은 `Cursor + Claude Code` 하이브리드 전제를 유지합니다.
- 메인 스택은 `React`, `Vite`, `TypeScript`, `Supabase`, `Cloudinary`, `n8n`입니다.
- AI 파싱은 현재 `Gemini` 메인을 기준으로 두되, 장애 대응을 위한 폴백 구조를 유지합니다.

## 7. 성공 기준

- 상담/견적/시공 사례 흐름이 끊기지 않고 연결된다.
- 직원이 사진과 사례를 빠르게 찾아 상담에 활용할 수 있다.
- 고객 공개 경험이 `쇼룸형 홈페이지 -> 문의 -> 후속 정보`로 자연스럽게 이어진다.
- 자동화가 실무를 흔들지 않고 보조한다.
