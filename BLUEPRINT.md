# 프로젝트 설계도 (BLUEPRINT)

> 이 문서는 FINDGAGU OS의 `현재 구현 기준서`입니다.  
> 운영 철학은 `CONTEXT.md`, 최근 결정은 `JOURNAL.md`, 과거 상세 설계는 `docs/history/blueprint-history.md`를 참조합니다.

## 1. 현재 제품 구조

### 공개 표면

- `쇼룸형 홈페이지`: 고객 공개 메인 경험으로 정의된 표면. 다만 현재 홈페이지 작업은 보류 상태이며, 대표가 명시적으로 다시 열기 전까지 활성 작업 대상으로 보지 않습니다.
- `/public/showroom`: 로그인 없이 여는 `오픈쇼룸`
- `/contact`: 문의 전환
- 토큰 기반 외부 쇼룸: 메인 표면이 아니라 보조 링크

### 내부 표면

- `/showroom`: 상담 중 설명, 선별, 공유를 위한 `내부쇼룸`
- `상담 관리`: 작업형 대시보드
- `이미지 자산 관리`: 사례/상담/콘텐츠용 공통 허브
- `실측 관리`: 내부 기술 자산 모듈
- `견적 관리`: 발행, 저장, 비교, 이력 관리
- 관리자 도구: 마이그레이션, 아카이브, 테스트 콘솔

## 2. 현재 우선순위

1. 이미지 자산 허브 안정화
2. 쇼룸형 홈페이지 고도화
3. 채널톡 후속 정보 레이어 정리
4. 공개 동선과 내부 운영 동선 분리 유지

## 3. 핵심 도메인 모델

### 상담

- 상담은 운영의 중심 엔티티입니다.
- 상태는 `접수 / 견적 / 진행 / 완료 / AS / 무효 / 거절` 7종을 기준으로 봅니다.
- 상태는 실무자의 판단값이며, 자동화는 보조 신호만 제공합니다.

### 사례

- 사례는 사진, 현장 문맥, 전후 비교, 예산/해결 포인트를 함께 가지는 자산입니다.
- 쇼룸과 채널톡 후속 정보는 같은 사례 문맥을 공유해야 합니다.

### 이미지 자산

- 이미지 자산은 단순 저장소가 아니라 `쇼룸`, `상담 보조`, `콘텐츠 자동화`의 공통 소스입니다.
- 공개 노출용 사례와 내부 기술 자산은 섞지 않습니다.

### 실측 자료

- 실측 PDF와 치수 메모는 상담 타임라인과 분리된 내부 기술 자산입니다.
- 상담 상세에서는 진입점만 제공하고, 실제 열람과 수정은 전용 모듈에서 처리합니다.

## 4. 활성 구조 원칙

### 공개/내부 분리

- 공개 가능한 라우트와 내부 운영 라우트는 명확히 분리합니다.
- 최소 공개 범위는 `쇼룸`, `문의`, 공개 공유 링크 수준입니다.

### 정보 계층화

- 상담 타임라인에는 흐름과 판단만 남깁니다.
- 이미지, 실측, 대용량 문서, 전체 견적 이력은 전용 모듈로 분리합니다.

### 사례 중심 설득

- 쇼룸은 단순 이미지 모음이 아니라 고객이 사례를 좁혀 가는 탐색 구조여야 합니다.
- `Before/After`는 일반 사례와 분리된 특수 문맥으로 다룹니다.
- 쇼룸 유입 문의에는 반드시 문맥 정보가 남아야 합니다.

## 5. 핵심 화면 규칙

### 상담 관리

- 상담 관리는 `작업형 대시보드`입니다.
- 데스크톱에서는 우측 상세 패널을 기준판처럼 유지합니다.
- 필터/정렬 변경 시 현재 선택 카드는 가능한 한 유지합니다.
- 상태 변경으로 탭이 바뀌어도 해당 카드가 포함된 페이지를 추적해야 합니다.
- 목록 기본 페이지 크기는 40개입니다.

### 이미지 자산 관리

- 상단 제어는 `검색 + 업로드 + 상담용 보기 + 업종/현장/제품/색상 필터` 구조를 유지합니다.
- 좌측은 필터가 아니라 `결과 네비게이션`입니다.
- 카드 우상단 배지 우선순위는 `Before/After -> 상담용 -> 대표지정`입니다.

### 쇼룸

- `쇼룸형 홈페이지`는 메인 공개 경험 정의를 유지하되, 현재 구현 작업은 보류 상태로 둡니다.
- `내부쇼룸`과 `오픈쇼룸`은 같은 데이터 문맥을 보더라도 목적이 다릅니다.
- `내부쇼룸`은 상담 중 설명과 내부 공유에 집중하는 화면입니다.
- `오픈쇼룸`은 로그인 없이 외부에 공개 가능한 쇼룸입니다.
- 로컬 개발에서 `내부쇼룸` 수정이 필요할 때는 로그인 플로우 대신 현재 우회 진입 경로를 사용합니다.
- 로그인 후 로컬 작업 화면 대신 배포된 Vercel 주소로 이동하는 문제가 해결되기 전까지는 이 우회 방식을 유지합니다.
- 공감카드와 검색어는 분리합니다.
- `Before/After` 자산은 기본 갤러리에 섞지 않습니다.
- 문의 전환 시 `showroom_context`, `showroom_entry_label`을 남깁니다.

## 6. 자동화 구조

### 운영 중인 흐름

- Google Chat 스페이스 생성 -> GAS AutoAddBot -> n8n -> Supabase 상담카드 생성
- 이미지 자산 -> 쇼룸/상담/콘텐츠 재활용

### 보류 중인 흐름

- Google Chat 실시간 견적 자동화는 구현 자산은 유지하되 운영 기본 흐름에서는 제외합니다.

### 채널톡 방향

- 초기 도입은 AI 대화형보다 폼 기반 질문 수집을 우선합니다.
- 채널톡은 쇼룸 링크 재전송보다 `관심 사례 기준 후속 정보`를 제공하는 레이어로 봅니다.

## 7. 기술 전제

- 프론트엔드: `React`, `Vite`, `TypeScript`
- 백엔드: `Supabase`
- 자동화: `n8n`
- 이미지 자산: `Cloudinary + Supabase`
- 개발 방식: `Cursor + Claude Code` 하이브리드
